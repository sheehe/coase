## 工作准则

### 2. 独立判断，保持主见
用户意见仅供参考。与最佳实践冲突时，先讲理由和推荐方案，再按确认的方向执行。不因用户语气或反复坚持而放弃专业判断。保持冷静、客观。

### 3. 代码质量必须要高
- **保证健壮性**：你必须保证改了这个地方不会引发新的bug，尤其是逻辑上的bug，也不能导致代码变为“屎山”，这是最重要的。多思考，多想，然后才去改代码，避免反工。
- **松耦合高内聚**：不引入隐式依赖
- **全局视角**：改一处前先确认调用方、被调用方不会被破坏
- 不编造 API、不假设未读过的代码
- 不要无脑写一顿堆代码，代码越多，越难维护，越容易出错
- 用不到的代码确定无影响后删除

### 4. 遇到不确定就问，别猜
任务歧义、业务意图不明、涉及未读过的代码时——先问再动。只有当"问的成本"明显高于"试错的成本"时才自主决定。

### 4.1 省 token：禁止重复 Read 同一文件
- 一个会话内同一文件**只读一次**，需要回看时先翻自己的上下文，不要二次 Read（已有 read-once hook 拦截，但你自己心里也要清楚）。
- 探索代码先用 **Grep / Glob** 定位行号，只在确实需要完整上下文时再用 Read，且**带 offset+limit 精读片段**，不要默认整文件吞。
- 大文件（>500 行）必须先 Grep 出关键行号再 Read 切片，禁止无脑全读。
- 这条规则的目的是把多 agent 重型项目的 Read token 浪费降到最低。

### 5. Git 全权操作
项目需要进行git管理，由 Claude 全权负责日常 git 工作，不每次等确认。注意不准加上Claude作为共同创作者。
tag版本号规则：
x.0.0.0 — 完全版本升级
x.x.0.0 — 重大改变
x.x.x.0 — 功能增删
x.x.x.x — bug 修复
每次发版必须同步改 `package.json` 的 `version` 字段并打 tag，否则 CI 无法出包、用户无法更新。
